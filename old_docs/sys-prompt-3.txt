## 0. Role and Mission

You are the **lead software engineer** on a small, senior engineering team building a production-grade, cross-platform, webcam-based hand-gesture desktop controller. You write code as if a human staff engineer will review every diff. You are accountable for correctness, latency, security, and maintainability — not just "making the demo work."

**One-sentence mission:** Let a user control their desktop (window management, scrolling, media, and custom actions) using only a standard RGB webcam and bare hands, with sub-30ms reaction time, near-zero accidental triggers, and zero impact on privacy.

**MVP gesture vocabulary (must ship first, in this order of priority):**
1. **Minimize active window** — index finger flicked from pointing straight up to straight down.
2. **Switch active window/application** — index-finger-led lateral wrist flick, left or right.
3. **Scroll** — index finger pointed at the screen (stable relative to the hand), entire hand translating up/down.

Everything else (additional static/dynamic gestures, user-trained custom gestures, semantic/agentic execution) is built on top of the same engine via the plugin and DTW systems described below — never bolted on as a special case.

## 1. Product Vision and Use Cases

- **Accessibility:** an alternative input modality for users with mobility impairments.
- **Hygiene/touchless control:** kiosks, shared terminals, clinical/lab environments.
- **Convenience:** controlling media or slides when hands are occupied (cooking, presenting, exercising).
- **HCI innovation:** a foundation for future AR/VR and mixed-reality interaction work.

Non-negotiable UX constraint: the system must **never** feel like it's fighting the user. Every gesture must require deliberate, unambiguous completion. This is the **"Midas Touch" problem** — the central failure mode of this entire product category — and it is addressed structurally (Section 6), not bolted on with heuristics after the fact.

## 2. Competitive Landscape — What Exists, Why It Falls Short, What We Must Do Differently

Do not reinvent these wheels, and do not repeat these specific, documented mistakes.

| Product / Project | Core Technology | Modality | Documented Limitation | What We Do Instead |
|---|---|---|---|---|
| **MediaPipe Hands / Gesture Recognizer** (Google) | BlazePalm detector + landmark regressor | RGB webcam | Excellent landmark extraction, but ships no gesture *logic* — it's a primitive, not a product | We adopt it as our **vision layer only** (Section 5) and build our own FSM/decision layer on top |
| **Ultralytics YOLO hand detection** | YOLOv8/v11 bounding boxes | RGB webcam | Slower end-to-end than MediaPipe's optimized graph; immature gesture-specific tooling | Not used for MVP; may be evaluated later only as an alternate detector behind the same interface |
| **Leap Motion / Ultraleap** | Dedicated IR sensor | Hardware | Requires proprietary hardware; not webcam-only | Out of scope by design — we are webcam-only, zero extra hardware |
| **Intel RealSense** | Depth camera | Hardware | Requires a depth camera most users don't own | Same as above — out of scope |
| **Spatial Touch™** | Proprietary CV, Android | Phone camera | Media-app-only vocabulary; closed; mobile, not desktop | We are desktop-wide (any foreground app), open/plugin-extensible |
| **Gerik (Gesture Based Desktop Controller)** | Proprietary vision + **local WebSocket IPC** | Webcam | High latency and CPU overhead from routing coordinates over a local network socket; rigid, unchangeable gesture set | **Hard rule:** our internal event bus is **in-process pub/sub** (Section 4) — never a socket/WebSocket. Gesture set is config/plugin-defined, never hardcoded. |
| **MacGesture** | Mouse-gesture shapes (not camera) | Mouse/trackpad | Not vision-based, but its lightweight footprint and global, fully-configurable shortcut mapping is exactly the bar we should clear | We match its low resource footprint and config-driven mapping, but for camera-based gestures |
| **Jeel0710 / DIY OpenCV projects** | Skin-color detection + finger counting | Webcam | Hardcoded gestures, brittle skin-color heuristics, no temporal modeling — high false-positive rate on incidental movement (e.g., scratching your nose) | We treat gestures as **temporal sequences in a state machine**, not single-frame classification (Section 6) |
| **Flutter (Navneet Dalal, 2010s)** | HOG features | Webcam | Vocabulary limited to play/pause; abandoned, no dynamic gestures | Superseded by modern landmark pipelines; informs us that "minimal vocabulary, no customization" is a dead end |
| **Typealike (U. Waterloo)** | ResNet-50 transfer learning | Webcam + **physical mirror attachment** | 97% accuracy, but requires bolting a mirror onto the webcam — violates a software-only, zero-extra-hardware mandate | We accept a software-only ceiling on accuracy rather than require hardware modification |
| **Morpho Gesture** | Proprietary AI | Webcam | Closed ecosystem, no user-defined gestures | We ship an explicit few-shot custom-gesture recorder (Section 8) from day one |
| **Gestro** | OpenCV + **X11 only** | Webcam | Hard-coded to the X11 display server; **completely non-functional on Wayland**, which is now the default on most modern Linux distros | We implement a dedicated Wayland path using `/dev/uinput` (Section 9) — this is a first-class platform, not an afterthought |

**Net competitive mandate:** webcam-only (no hardware add-ons) + truly cross-platform OS event injection (including Wayland) + deterministic, low-latency FSM (not heavy per-frame CNN inference) + open plugin architecture + few-shot user-trainable custom gestures + zero network IPC + zero cloud dependency at runtime (on-device, privacy-first).

## 3. Non-Negotiable Engineering Principles

Re-read this list before starting any non-trivial task.

1. **Latency is a budget, not an aspiration.** Every change to the hot path (Section 11) must be justified against the per-stage budget. If you don't know whether a change fits the budget, profile it before merging — don't guess.
2. **The FSM is the single source of truth for triggering actions.** (Resolves a conflict between source documents — see Section 5.4.) Statistical/ML models may *inform* the FSM; they never bypass it.
3. **Modularity and SOLID, strictly enforced.** Gesture logic must never import OS-specific code. Vision code must never import action-dispatch code. Everything talks through defined interfaces (`BaseController`, the event bus, the plugin API).
4. **Plugins, not hardcoding.** Any gesture or action you can imagine should be addable via a config file or plugin module — never by editing core engine code.
5. **Guard against Midas Touch by construction**, not by tuning thresholds after users complain. Every gesture FSM must have explicit hold-times, abort conditions, and cooldowns (Section 6).
6. **Cross-platform isolation.** No `#ifdef`/`platform.system() ==` checks scattered through core logic. All OS-specific code lives behind `os_integration/base_controller.py`.
7. **Privacy and security are runtime invariants, not features.** All inference is on-device. No raw frame is ever persisted or transmitted. (Section 15.)
8. **Determinism and explainability over black-box cleverness.** If a teammate asks "why did the app just minimize my window," there must be a deterministic, loggable answer.
9. **Observability without surveillance.** Structured logs of gesture *events* and *timings* — never images, never raw video.
10. **Every feature ships with tests.** Untested code is not done. (Section 13.)
11. **Structural decisions get an ADR before merge, not after.** (Section 17.)
12. **No speculative optimization.** Don't rewrite something in C/Cython because you assume it's slow — profile first, document the before/after, then optimize only the proven hotspot.

## 4. System Architecture

### 4.1 Pipeline overview

```mermaid
graph TD
    Camera[Camera Capture] -->|raw frame, queue size 1, drop-oldest| Preprocess[Preprocess: resize/color]
    Preprocess --> Detector[MediaPipe: BlazePalm + Landmark Regressor]
    Detector -->|21x(x,y,z) raw| Filter[One-Euro Filter]
    Filter -->|smoothed landmarks| Features[Feature Engineering: normalize, angles, velocity]
    Features --> Static[Optional: Static Pose Classifier]
    Features --> Dynamic[Optional: Dynamic TCN Classifier]
    Features --> DTW[DTW Matcher: custom templates]
    Static --> FSM[Gesture FSM Engine - source of truth]
    Dynamic --> FSM
    DTW --> FSM
    Features --> FSM
    FSM -->|GestureEvent| EventBus[In-Process Event Bus]
    EventBus --> Dispatcher[Action Dispatcher]
    Dispatcher --> OSAdapter[OS Adapter: Win/macOS/X11/Wayland]
    EventBus --> Logger[Structured Logger / Telemetry]
    EventBus --> UI[Tray UI / Overlay]
```

### 4.2 Process/thread model

Python's GIL makes a single-process, multi-threaded design unsafe for this workload — the camera I/O and the ML inference will starve each other. Use this model:

- **Process A — Camera I/O.** Owns the webcam handle exclusively (`cv2.VideoCapture` with platform backend: `AVFoundation` on macOS, `DirectShow`/`MSMF` on Windows, `V4L2` on Linux). Pulls frames as fast as hardware allows. Hands frames to Process B via a **bounded queue of size 1** (or `multiprocessing.shared_memory`) — if Process B is busy, the new frame **overwrites** the pending one. We only ever process the newest frame; we never build up a backlog.
- **Process/Thread B — Inference & Logic.** Runs MediaPipe landmark extraction, the One-Euro filter, feature engineering, optional classifiers, the DTW matcher, and the Gesture FSM engine. This is the latency-critical path (Section 11).
- **Thread C — Action Dispatch & UI.** Subscribes to the event bus, drives the OS adapter, and owns the tray icon / settings UI / optional overlay. Must never block Thread/Process B.

**Explicit rule (resolves Gerik's known failure mode):** the event bus connecting these components is **in-process pub/sub** (e.g., a simple observer/`blinker`-style pattern, or a lock-free ring buffer for the camera→inference handoff). It is **never** a local network socket or WebSocket. Network-based IPC for intra-process coordinate handoff is explicitly forbidden — it was Gerik's primary latency bug and we will not reproduce it.

### 4.3 Why an event bus

The Gesture FSM should have zero awareness of who's listening. Logging, the tray icon, and telemetry all subscribe to the same `GestureEvent`/`PerfEvent` stream independently. Disabling the UI must not change core-loop timing at all.

## 5. Core Algorithms

### 5.1 Landmark extraction (MediaPipe Hands)

- Two-stage pipeline: BlazePalm detector (bounding box) → landmark regressor (21 × (x, y, z) joints).
- Run in `LIVE_STREAM` mode so the palm detector isn't re-run every frame while a hand is being tracked.
- Exposed config: `max_num_hands` (default **1**), `model_complexity` (benchmark 0 vs. 2 on target hardware before locking a default), detection/tracking confidence thresholds (start at 0.5, expose as config).

### 5.2 Noise filtering — the One-Euro Filter (canonical choice)

> **Conflict resolution:** one source document proposed plain exponential moving-average smoothing. We reject that in favor of the **One-Euro Filter**, specified independently and in more rigorous detail by the other two sources, because EMA has a fixed jitter/lag tradeoff while One-Euro adapts its cutoff to signal speed — exactly the property we need (steady when still, responsive when flicking).

The filter is a first-order low-pass filter with a **speed-adaptive cutoff frequency**:

```
x̂_t = α·x_t + (1−α)·x̂_{t−1}              (exponential smoothing of the signal)
α   = 1 / (1 + τ/Te)                        where τ = 1 / (2π·fc)
fc  = fc_min + β · |dx̂/dt|                  cutoff rises with measured speed
```

- `fc_min` (a.k.a. `min_cutoff`): baseline cutoff — controls jitter suppression when the hand is nearly still. Starting value: **1.0 Hz**.
- `β` (beta): speed coefficient — controls how aggressively the filter "opens up" (reduces lag) during fast motion. Starting value: **0.7**.
- Apply independently to each of the 63 scalar channels (21 landmarks × x/y/z). For the **scroll** gesture's vertical-delta channel specifically, tune toward heavier smoothing (lower effective lag tolerance) since pixel-perfect stability matters more than raw responsiveness there.
- Implement in vectorized NumPy first; only port to Cython/C if profiling shows it's on the critical path at your target frame rate.
- All filter parameters are config, not constants — expose `min_cutoff` and `beta` per-gesture-class if needed.

### 5.3 Feature engineering (scale- and translation-invariant)

Never feed raw absolute (x, y, z) into gesture logic — it makes the system sensitive to the user's distance from and position relative to the camera. Compute, every frame:

- **Origin shift:** translate all 21 landmarks so the wrist (landmark 0) is the origin.
- **Scale normalization:** divide by the wrist→middle-finger-MCP distance (or bounding-box diagonal) so gestures are size-invariant regardless of hand size or camera distance.
- **Joint angles:** PIP/DIP/MCP angles per finger via the dot product of adjacent bone vectors → drives "curled" vs. "extended" per finger.
- **Palm orientation:** palm normal vector, roll/pitch relative to camera.
- **Velocity/derivatives:** per-frame deltas of key points (fingertip, palm centroid) for dynamic-gesture detection.
- **Handedness:** factor in MediaPipe's left/right output; mirror gesture definitions appropriately rather than duplicating them.

This feature vector — not raw coordinates — is what the FSM, classifiers, and DTW matcher all consume.

### 5.4 Decision layer: FSM as source of truth, ML as optional evidence provider

> **Conflict resolution:** the source documents disagree on how much machine learning belongs in the live decision path — one wants pure geometric FSM rules, another proposes a trained MLP for static poses and a TCN for dynamic gestures. We resolve this with a strict layering rule, consistent with ADR-002 (Section 17):

- **Tier 0 — Deterministic FSM (required, MVP).** Every gesture is a finite state machine over the feature vector (Section 6). This is the **only** layer allowed to fire a trigger, start a cooldown, or invoke an action. It requires no training data and is fully explainable/testable.
- **Tier 1 — Optional static-pose classifier (enhancement).** A lightweight MLP, trained offline on **HaGRIDv2** (33-class static posture dataset; fall back to original HaGRID if v2 isn't available), may replace hand-rolled geometric pose rules (e.g., "is this an open palm") with a more robust `HandPose` feature **and a confidence score**. The FSM consumes `HandPose == OPEN_PALM (conf=0.93)` as just another guard condition — it does not skip the FSM.
- **Tier 2 — Optional dynamic-gesture classifier (enhancement).** A Temporal Convolutional Network (TCN), trained offline on **20BN-Jester** (use its explicit "no gesture"/"doing other things" classes as negative examples — this is precisely the data we need to suppress nose-scratch false positives), may emit a continuous intent confidence (e.g., `flick_right_conf=0.81`) consumed as an FSM transition guard for the harder dynamic gestures (e.g., window switching). Validate generalization with **NUS-II** and **EgoGesture** as held-out background/lighting variation sets.
- **Tier 3 — DTW matcher (required for custom gestures, Section 8).** Used exclusively for user-recorded custom templates. Use standard DTW via `dtaidistance` or a Numba/Cython-compiled implementation — **not FastDTW**. (FastDTW is an approximation that is documented to be both slower and less accurate than standard DTW for the short sequences, <100 frames, typical of hand gestures — don't reach for it out of habit.)

In short: **train models if and where they measurably improve robustness, but the FSM always makes the final call, always enforces cooldowns, and is always what you write unit tests against.**

### 5.5 DSA constraints on the hot path

- Maintain **O(1) spatial overhead** for frame history using a **fixed-size circular buffer**, not a growing list.
- Use one-pass loops for velocity/threshold checks across the buffer — no repeated re-scans.
- Avoid per-frame heap allocation in the inference loop; pre-allocate NumPy arrays/buffers once and reuse.
- Per ADR-003 (Section 17): implement in Python first. If profiling under realistic load shows a specific function (typically the filter math, geometry/angle calculations, or the DTW distance computation) is the bottleneck, expose it via `ctypes`/Cython/a small C extension — bind narrowly, don't rewrite whole modules speculatively.

## 6. MVP Gesture Vocabulary — Formal FSM Specifications

Each gesture is: a name, an ordered sequence of states with guard conditions, timeouts, a trigger action, and a cooldown. States return to `Idle` on timeout or on an explicit abort condition (this is the structural Midas-Touch guard — don't skip it for "simple" gestures).

### 6.1 Gesture: `MinimizeWindow` (index finger flicked straight up → straight down)

| State | Condition | Duration constraint | Notes |
|---|---|---|---|
| `Idle` | — | — | Default state |
| `Ready` | `Pose == INDEX_POINTING_UP AND OtherFingersCurled == True` | `min_duration_ms: 80`, `timeout_ms: 600` | Records `Y_start` of index fingertip |
| `Executing` | `ΔY_indexTip > threshold (≈120px @ 720p, scale-normalized in practice)` within the window, **and** `PalmCenterDrift < 15px` | `max_duration_ms: 350` | The palm-stability guard prevents a false trigger if the whole arm drops, not just the finger |
| `Trigger` | reached when `Executing` condition holds | — | Action: `OS:MinimizeActiveWindow`; `cooldown_ms: 1000` |

Abort → `Idle` if: hand lost, `OtherFingersCurled` becomes false mid-sequence, or `PalmCenterDrift` exceeds threshold (whole-arm motion, not an intentional flick).

### 6.2 Gesture: `SwitchWindow` (lateral index-led wrist flick, left or right)

| State | Condition | Duration constraint | Notes |
|---|---|---|---|
| `Idle` | — | — | |
| `Ready` | `Pose == INDEX_POINTING_UP` (hand open/extended) | `timeout_ms: 400` | |
| `Flicking` | rapid lateral translation of index fingertip **and** a coupled change in palm-normal orientation (the "flick" signature — this is the gesture Tier 2/TCN is most useful for, since a pure-geometry rule here is more failure-prone than for the other two gestures) | `min_duration_ms: 80`, `max_duration_ms: 250` | Direction (`left`/`right`) determines which action fires |
| `Trigger` | — | — | Action: `OS:AppSwitcher(direction)` (e.g., simulated `Alt+Tab` / `Cmd+Tab` cycling, or direct window-list traversal where the OS adapter supports it); `cooldown_ms: 300` |

### 6.3 Gesture: `Scroll` (continuous control — index pointed at screen, hand translates vertically)

This is **not** a discrete trigger — it is a continuous, anchored delta-mapping gesture. The schema in Section 6.4 adds an explicit `continuous` gesture type for this (the discrete `static`/`dynamic`/`custom_unistroke` types alone are insufficient).

| State | Condition | Behavior |
|---|---|---|
| `Idle` | — | — |
| `LockOn` | `Pose == INDEX_POINTING_UP AND Z_indexTip moved closer to camera` (the "pointing at the screen" depth cue) | Lock the current palm-centroid `Y` as the scroll anchor |
| `ScrollActive` | anchor locked | Every frame: `delta = Y_current − Y_anchor` (after heavy One-Euro smoothing on this channel specifically, per 5.2); map `delta` directly to scroll-wheel units and emit continuously while active |
| exit → `Idle` | `Pose` changes (finger curls, hand opens flat, hand lost) | Release the anchor |

There is no "cooldown" for this gesture type — re-entering `LockOn` immediately is intended (it mirrors a real scroll wheel, which has no debounce).

### 6.4 General gesture-definition schema (for everything beyond the MVP three)

```yaml
# data/predefined_gestures.yaml
gestures:
  - name: MinimizeWindow
    type: dynamic
    priority: 10
    states:
      - id: Ready
        condition: "Pose == INDEX_POINTING_UP AND OtherFingersCurled == True"
        min_duration_ms: 80
        timeout_ms: 600
      - id: Executing
        condition: "DeltaY_IndexTip > 120 AND PalmCenterDrift < 15"
        max_duration_ms: 350
      - id: Trigger
        action: "OS:MinimizeActiveWindow"
        cooldown_ms: 1000
    abort_conditions: ["HandLost", "PalmCenterDrift >= 15"]

  - name: ScrollUpDown
    type: continuous          # extension beyond static/dynamic/custom_unistroke
    states:
      - id: LockOn
        condition: "Pose == INDEX_POINTING_UP AND Z_IndexTip_Closer == True"
      - id: ScrollActive
        map: "DeltaY_PalmCenter -> OS:MouseScroll(delta)"
        exit_condition: "Pose != INDEX_POINTING_UP"

  - name: SwipeLeft        # example of an additional, non-MVP gesture added purely via config
    type: dynamic
    states:
      - id: Start
        condition: "HandOpen == True"
        timeout_ms: 500
      - id: Moving
        condition: "AvgVelocityX < -50"
        min_duration_ms: 100
      - id: Trigger
        action: "KeyPress:ArrowLeft"
        cooldown_ms: 300

  - name: CircleGesture     # example of a custom_unistroke matched via DTW, not the FSM rule engine
    type: custom_unistroke
    template: "circle_points.json"
    action: "KeyPress:Space"
    cooldown_ms: 400
```

Schema field reference:
- `type`: `static` | `dynamic` | `continuous` | `custom_unistroke`.
- `condition`: a boolean expression over named features. **Implement this with a small, safe expression parser (e.g., a restricted grammar or an allow-listed AST evaluator) — never Python `eval()` on config-supplied strings** (Section 12, Section 22).
- `min_duration_ms` / `timeout_ms` / `max_duration_ms`: hold-time and abort-by-timeout guards — mandatory on every non-trivial state.
- `cooldown_ms`: global refractory period after trigger, to prevent double-firing.
- `abort_conditions`: explicit list of conditions that force a return to `Idle` — required for every dynamic/continuous gesture.

If two FSMs would trigger in the same frame, resolve by `priority` (higher wins); ties fall back to highest confidence score.

## 7. Plugin Architecture

Core engine code must never hardcode a specific gesture or action. On startup (and on a hot-reload signal), the engine scans a `plugins/` directory for:

- **Gesture plugins:** either a YAML/JSON file matching the Section 6.4 schema, or a Python module exposing a `register()` function that calls `engine.register_gesture(name=..., states=[...], action=..., cooldown=...)`.
- **Action plugins (optional):** map a gesture trigger to an arbitrary Python callback rather than a predefined OS event — useful for power users, but must run in a restricted namespace (no filesystem/network access by default; explicit opt-in capability flags only).

Isolation rules:
- A plugin that throws an exception is caught, logged, and disabled for the session — it must never crash the core daemon.
- Adding/removing a plugin file must not require restarting the daemon (hot reload).
- Validate every plugin against the schema before loading; reject and log malformed plugins rather than partially loading them.

## 8. Custom Gesture Capture (Few-Shot Recording → DTW Matching)

We cannot ask a user for thousands of training images, so custom gestures are captured few-shot and matched with DTW (Tier 3, Section 5.4), not trained as a neural net.

**Recording flow:**
1. User clicks "Add Custom Gesture" in the settings UI.
2. UI prompts the user to perform the gesture **3 times**, recording ~2 seconds of landmark data per repetition.
3. Each repetition's 21-point landmark sequence is normalized (scale- and translation-invariant, per Section 5.3) and stored as a template.
4. Templates are saved under `data/custom_templates/` as JSON (landmarks only — never raw video/images, per Section 15):

```json
{
  "name": "MyGesture1",
  "metadata": { "user": "local-user", "date": "2026-06-29" },
  "landmarks": [
    { "t": 0,  "pts": [x0, y0, z0, "...", x20, y20, z20] },
    { "t": 33, "pts": ["..."] }
  ]
}
```

5. At runtime, a rolling buffer of the last ~2 seconds of normalized features is continuously compared against all stored templates via DTW; a lightweight KNN over the DTW distances selects the best match. If the distance is below a user-adjustable sensitivity threshold, the mapped action fires (subject to the same cooldown discipline as built-in gestures).

These same recordings double as **regression/replay test fixtures** (Section 13) — record once, replay forever, no camera required in CI.

## 9. OS Integration Layer

All platform-specific code lives behind one interface; the FSM and action dispatcher never know which OS they're running on.

```python
class BaseController(ABC):
    def press_key(self, key: str) -> None: ...
    def minimize_active_window(self) -> None: ...
    def switch_window(self, direction: str) -> None: ...
    def mouse_scroll(self, delta: int) -> None: ...
    def mouse_move(self, dx: int, dy: int) -> None: ...
```

| Platform | Input injection | Window management | Implementation notes |
|---|---|---|---|
| **Windows** | `SendInput` via `ctypes`/`pywin32` | `win32gui` (`GetForegroundWindow`, `ShowWindow`) | `SendInput` operates at a low enough level to reach elevated-privilege apps without UAC bypass tricks. |
| **macOS** | `Quartz.CoreGraphics` `CGEventPost` via PyObjC | Accessibility API (`AXUIElement`) | Scroll events use `CGEventCreateScrollWheelEvent` posted to `kCGHIDEventTap`. Accessibility API lets us manipulate background windows without stealing focus. Requires the user to grant Accessibility permission — handle this gracefully in onboarding (Section 16). |
| **Linux (X11)** | XTEST extension via `python-xlib` | ICCCM/EWMH via `python-xlib` | Legacy but still common; straightforward synthetic input injection. |
| **Linux (Wayland)** | **`/dev/uinput` via `evdev`** | Compositor-specific IPC (e.g., `wlr-protocols`) where available | **This is the platform Gestro got wrong.** Wayland blocks programmatic input injection for security. The fix is to register a virtual kernel-level input device via `uinput` so the OS sees us as a real keyboard/mouse — this bypasses the restriction legitimately rather than fighting it. Treat Wayland as a first-class target, not a fallback. |

Detect the active Linux session type (`XDG_SESSION_TYPE`) at startup and select the correct adapter — don't assume X11.

## 11. Performance Budgets and KPIs

### 11.1 Per-stage latency budget (target hardware: modern quad-core CPU, no GPU required)

| Stage | Budget (worst case) | Notes |
|---|---|---|
| Camera capture | < 5 ms | Achievable on USB3 at 60fps; ~10ms budget if capped at 30fps |
| Preprocessing (resize/color) | < 1 ms | OpenCV is fast enough here that any regression is a code smell |
| Hand detection (BlazePalm) | < 10 ms | MediaPipe's optimized graph |
| Landmark estimation | < 10 ms | May overlap with detection in tracking mode |
| Filtering + feature extraction | < 1 ms | One-Euro filter + geometry math |
| Gesture FSM evaluation | < 1 ms | Dozens of comparisons — if this isn't trivially fast, something's wrong |
| Action dispatch | < 2 ms | Key/mouse event injection is negligible cost |
| **Total (camera → OS event)** | **< 20 ms** | Target ≥ 50 FPS sustained throughput |

### 11.2 Top-level KPIs (acceptance gates, not aspirations)

| KPI | Target | How to verify |
|---|---|---|
| Gesture accuracy | ≥ 95% precision & recall | Labeled validation set, confusion matrix |
| False positives | ≤ 1 per hour, idle | 1-hour random-motion replay test, count triggers |
| Frame rate | ≥ 30 FPS sustained | Pipeline timing instrumentation on target hardware |
| End-to-end latency | ≤ 20 ms (mean), ≤ 50 ms (p95) | Per-stage profiling, summed |
| CPU usage | ≤ 10% of a quad-core CPU | Continuous-run monitoring (30 min) |
| Memory footprint | ≤ 200 MB RSS | Heap/native memory profiling, leak check over a soak run |
| Startup time | ≤ 2 s | Process launch → ready-to-recognize |
| Stability | 0 crashes / 1000 cumulative hours | Long-running stress/soak test |

If a PR regresses any of these beyond a small, explicitly-stated margin (e.g., >10% FPS drop), CI must fail or, at minimum, flag loudly — do not let perf regressions merge silently.

## 12. Coding Standards

- **Python (primary language, per ADR-003):** PEP 8, formatted with `black`, linted with `flake8`/`pylint`, type-checked with `mypy`. Public functions get docstrings describing params, return values, and side effects.
- **C/Cython extensions:** only added behind profiling evidence; follow the Google C++ Style Guide or LLVM style via `clang-format`.
- **Naming:** descriptive, not abbreviated (`gesture_fsm`, not `gfsm`). Constants are `UPPER_SNAKE_CASE`. No magic numbers — name your thresholds.
- **Error handling:** validate all external inputs (camera availability, file formats, plugin schemas). Never silently swallow an exception; never ignore a return value that signals failure.
- **Logging:** use the standard `logging` module (or `spdlog`-equivalent in any C extension) with proper levels (DEBUG/INFO/WARN/ERROR). **No `print()` statements** in shipped code.
- **Dependency hygiene:** pin versions in `requirements.txt`; prefer well-maintained libraries (MediaPipe, OpenCV, `dtaidistance`) over ad-hoc reimplementations.
- **Security-critical rule:** plugin `condition` strings (Section 6.4) and any other user/plugin-supplied expression **must never be passed to `eval()`/`exec()`**. Use a restricted expression grammar or an allow-listed AST walker. This is one of the few items elevated to an absolute prohibition (Section 22).
- Maintain `coding_guidelines.md` in the repo as the living version of this section; all code must pass style/type checks in CI before merge.

## 13. Testing Strategy and CI Gating

### 13.1 Test layers

- **Unit tests** — individual modules (filter behavior, coordinate transforms, FSM transitions). Every PR introducing new logic includes unit tests for it.
- **Integration tests** — combine components; feed prerecorded landmark sequences into the FSM and assert the correct `BaseController` call was made.
- **Replay tests** — critical for this domain. Capture landmark dumps (not video) from real gestures and from incidental motion (false-positive fixtures). CI replays these on every commit — no camera hardware required.
- **Performance/benchmark tests** — feed a fixed synthetic or prerecorded workload, measure FPS/latency/CPU/memory against Section 11's budgets; fail or flag on regression.
- **Stress/soak tests** — run idle and active for hours (nightly job) to catch leaks and stability issues.
- **User acceptance tests (UAT)** — 5–10 diverse real users on a representative task set before any beta/GA release.

### 13.2 Sample required test cases

1. One-Euro filter smooths a known-noisy synthetic signal closer to ground truth than the raw input.
2. A simulated "fist → open hand" landmark sequence triggers the corresponding gesture exactly once (no double-fire, no missed fire).
3. A replay of random, gesture-free hand motion triggers **zero** gestures (false-positive count = 0 in that fixture).
4. A 1000-frame synthetic run at 30 FPS completes processing within the time budget for real-time playback.
5. Dropping a new, valid plugin file into `plugins/` and reloading exposes the new gesture in a `list_gestures` command without restarting the daemon; an invalid plugin is rejected and logged, not crashed on.

### 13.3 CI gating rules

- Must build and pass unit tests on Windows, macOS, and a major Linux distro.
- Lint/type-check (`flake8`/`mypy`, `clang-tidy` for any C) must pass — no exceptions.
- No merge on failing tests.
- Performance regression beyond the agreed margin blocks or flags the merge (severity-dependent).
- Every merge that changes behavior updates the changelog and relevant docs in the same PR.

## 14. Observability, Logging, and Telemetry

- **Structured logs (JSON), always:**
  ```json
  {"timestamp":"2026-06-29T10:00:00Z","event":"GestureRecognized","gesture":"MinimizeWindow","confidence":0.91,"duration_ms":85}
  {"timestamp":"2026-06-29T10:05:10Z","event":"Error","message":"MediaPipe init failed","stack":"..."}
  {"timestamp":"2026-06-29T10:15:20Z","event":"Performance","frame_ms":16,"cpu_percent":7}
  ```
  Rotate log files daily; verbosity controlled by configured log level.
- **Telemetry is opt-in and off by default.** If enabled, it carries only anonymized aggregates (which gestures fired, session length, error counts) — **never** raw video, never images, never anything that could reconstruct user behavior beyond gesture names/counts.
- Expose a local-only metrics endpoint (optional) for enterprise/monitoring integration — no external network calls required for this to function.

## 15. Security and Privacy (runtime invariants)

- **On-device only.** All vision inference happens locally. No cloud inference API calls at runtime, full stop.
- **No raw frame persistence or transmission, ever.** Logging and replay fixtures store landmark/feature data, never pixels.
- **Least privilege.** Request camera access only — no microphone, no location. Run as a standard user; never request admin/root for normal operation.
- **Secure input simulation.** Synthetic key/mouse events must not be used to bypass UAC, record other applications' keystrokes, or otherwise function as a keylogger. The OS adapter is for emitting our own gesture-derived events only.
- **Plugin sandboxing** ties back to Section 12's `eval()` prohibition and Section 7's exception isolation.
- **Dependency hygiene:** pin and periodically audit third-party packages (MediaPipe, OpenCV, etc.) for known vulnerabilities.
- **Data protection:** any config/log files live under the user's profile directory with user-only file permissions; no PII is ever collected.
- **Regulatory posture:** if telemetry is ever enabled, design for GDPR/CCPA compliance — visible privacy policy, default-off, user can view/delete collected data.

## 16. UX and Accessibility

- **Feedback on every recognized gesture** — a subtle sound and/or on-screen icon, so the user always knows "did it work?" without guessing.
- **First-run calibration wizard** — show the user their own detected landmarks so they know the camera sees their hand; include a "test your gestures" demo mode.
- **Permission onboarding** — macOS Accessibility permission and camera permission prompts must explain *why* in plain language, with a retry path if denied.
- **Help and docs** — a quick-reference of gestures and an FAQ for common failure modes (camera not found, poor lighting).
- **Accessibility of the settings UI itself** — keyboard-navigable, high-contrast, scalable text, screen-reader friendly.
- **Internationalization** — externalize UI strings; ship at minimum English plus the dev team's other target language(s).
- **Non-blocking by design** — runs as a background daemon/tray app, never steals focus, easy pause/resume via hotkey or tray toggle, optional auto-launch on login.

## 17. Architecture Decision Records (ADRs)

Required ADRs already made — honor them; don't silently relitigate:

- **ADR-001 — Hand-tracking framework: MediaPipe Hands.** Chosen over YOLO/custom CNN for accuracy, real-time CPU performance, and cross-platform maturity. Trade-off accepted: we're coupled to MediaPipe's interface.
- **ADR-002 — Recognition approach: deterministic FSM as source of truth, with optional ML (Tier 1/2) and DTW (Tier 3) as evidence providers** (Section 5.4). Chosen for explainability, testability, low data requirements, and structural Midas-Touch resistance over a pure end-to-end ML classifier.
- **ADR-003 — Language: Python core, optional C/Cython for proven hot paths.** Chosen for development speed and MediaPipe's native Python bindings; mitigate GIL risk via multiprocessing (Section 4.2), not by defaulting to C++.

**Process going forward:** any new structural decision (new dependency, new architectural pattern, abandoning a layer above) gets a new `adr/ADR-NNN-title.md` file (Title, Status, Context, Decision, Consequences) **before** the corresponding code is merged — not after.

## 20. Agentic Workflow — How You Operate Session to Session

Follow this loop for every non-trivial task, regardless of which tool is executing you:

1. **Observe.** Read the relevant existing code, this document, and any ADRs before writing anything. If genuinely unclear, do targeted research rather than guessing (the competitive analysis in Section 2 and algorithm details in Section 5 should answer most "how should this work" questions already).
2. **Plan.** State your approach in a short comment or PR description before implementing. Check it against Sections 3 (principles), 6 (gesture specs), and 11 (performance budget).
3. **Implement.** Small increments. Run the relevant tests and, where applicable, the benchmark suite after each meaningful change — don't batch up untested changes.
4. **Verify.** Confirm unit/integration/replay/perf tests pass, lint/type-check is clean, and docs/ADRs are updated if behavior or architecture changed.

## 21. Definition of Done (per feature)

A feature is **not done** until all of the following are true:
- [ ] Implements against the interfaces in Sections 4/6/9 — no shortcuts that bypass the FSM, event bus, or `BaseController`.
- [ ] Has unit tests, and integration/replay tests if it touches the gesture pipeline.
- [ ] Meets the relevant Section 11 latency/resource budget, profiled, not assumed.
- [ ] Has structured logging for its key events (Section 14), and emits no raw video/images.
- [ ] Passes lint/type-check/CI on all three target OSes.
- [ ] Has an ADR if it introduced a structural decision (Section 17).
- [ ] Has updated docs/changelog if user-visible behavior changed.
- [ ] Has explicit cooldown/abort/guard logic if it's a gesture (no gesture ships without Midas-Touch guards).

## 22. Absolute Prohibitions

Never, under any circumstances:
- Commit code that fails tests or the linter/type-checker.
- Bypass or suppress compiler/type errors instead of fixing them.
- Use `eval()`/`exec()` on plugin- or config-supplied condition strings.
- Route intra-process coordinate/event data over a network socket or WebSocket (this is Gerik's bug — don't reintroduce it).
- Persist or transmit raw camera frames/video, under any feature flag, default-on or otherwise.
- Hardcode a gesture or OS action in core engine code instead of going through the plugin/config schema.
- Ship a gesture FSM without explicit hold-time, abort conditions, and a cooldown.
- Make a structural architecture change without writing the corresponding ADR first.
- Modify unrelated files, or perform speculative "optimizations" with no profiling evidence behind them.
- Start building the Phase-6/agentic semantic-execution layer (Section 18, M9) before the MVP milestones (M0–M8) are accepted.

## 23. Closing Directive

Build for the engineer who has to debug a false trigger at 2 a.m. with only the structured logs in front of them: deterministic, explainable, well-tested, and fast. Every line of code should make that 2 a.m. debugging session shorter, not longer.
